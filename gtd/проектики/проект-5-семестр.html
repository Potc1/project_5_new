Проект 5 семестр

Техническое задание на разработку аналога Google Meet "MeetSphere"

1. Введение

  1. Назначение документа
    1. Документ описывает требования к разработке веб-приложения для
    2. видеоконференций с базовым функционалом.
  2. Цель проекта
    1. Создание простого и удобного веб-приложения для проведения видеоконференций с поддержкой голосовой и видео связи, демонстрации экрана и текстового чата.

2. Общие требования

  1. Функциональные требования
    ✅ Видеоконференции до 12 участников
    ✅ Голосовая связь высокого качества
    ✅ Демонстрация экрана
    ✅ Текстовый чат
    ✅ Создание и присоединение к встречам по ссылке
  2. Нефункциональные требования
    Производительность: Задержка аудио/видео < 300 мс
    Совместимость: Chrome 90+, Firefox 88+, Safari 14+
    Безопасность: Шифрование трафика (WebRTC)
    Масштабируемость: Поддержка до 1000 одновременных сессий

3. Детальные требования к функционалу

  1. Система видеозвонков
    1. Видеосвязь
    • Поддержка разрешений до 1080p
    • Автоматическая подстройка качества под скорость сети
    • Возможность включения/выключения видео
    • Переключение между камерами
    1. Голосовая связь
    • Шумоподавление и эхоподавление
    • Автоматическая регулировка громкости
    • Возможность включения/выключения микрофона
    • Выбор устройств ввода/вывода аудио
  2. Демонстрация экрана
    • Демонстрация всего экрана
    • Демонстрация отдельного окна
    • Демонстрация вкладки браузера
    • Индикатор активной демонстрации
    • Переключение между демонстрацией и камерой
  3. Текстовый чат
    • Общий чат комнаты
    • Уведомления о новых сообщениях
    • История сообщений в течение сессии
    • Информация об отправителе и времени
  4. Управление встречей
    1. Создание встречи
      Генерация уникальной ссылки
      Настройка имени встречи
      Быстрый старт без регистрации
    2. Участники
      Отображение списка участников
      Индикация активности (микрофон/видео)
      Поиск по участникам
      Приглашение по ссылке

4. Техническая архитектура

  1. Фронтенд (клиентская часть)
    plaintext
    Технологии: React.js + TypeScript
    Библиотеки:
    • WebRTC API - видеосвязь
    • Socket.io-client - реальное время
    • Simple-Peer - P2P соединения
    • Styled Components - стилизация
  2. Бэкенд (серверная часть)
    plaintext
    Технологии: Node.js + Express + TypeScript
    Библиотеки:
    - Socket.io - WebSocket соединения
    - Simple-Peer - WebRTC сигналинг
    - UUID - генерация ID комнат
  3. База данных
    plaintext
    Тип: In-memory (Redis) для сессий
    Хранение:
    • Активные комнаты
    • Временные данные сессий
    • Не требует постоянного хранения

5. Интерфейс пользователя

  1. Основные экраны
    1. Главная страница
      Кнопка "Новая встреча"
      Поле для ввода кода встречи
      Краткая инструкция
    2. Страница предварительной настройки
      Превью видео и аудио
      Выбор устройств
      Ввод имени пользователя
      Настройки качества
    3. Экран конференции
      Сетка видео участников
      Панель управления
      Боковая панель чата
      Информация о встрече
  2. Элементы управления
    1. Нижняя панель управления
interface ControlPanel {
  microphone: { enabled: boolean, device: string }
  camera: { enabled: boolean, device: string }
  screenShare: boolean
  chat: boolean
  leaveCall: boolean
  settings: boolean
}
  1. Индикаторы состояния
  • Качество соединения
  • Активный спикер
  • Уведомления чата
  • Cтатус демонстрации экрана

6. WebRTC архитектура

  1. Сигналинг сервер
// Обработка WebRTC сигналов
interface SignalingEvents {
  'join-room': (roomId: string, userId: string) => void
  'offer': (offer: RTCSessionDescription, to: string) => void
  'answer': (answer: RTCSessionDescription, to: string) => void
  'ice-candidate': (candidate: RTCIceCandidate, to: string) => void
}
  1. P2P соединения
// Mesh сеть для малых групп (до 12 участников)
class PeerConnection {
  createOffer(): Promise<RTCSessionDescription>
  setRemoteDescription(description: RTCSessionDescription): Promise<void>
  addIceCandidate(candidate: RTCIceCandidate): Promise<void>
  addStream(stream: MediaStream): void
}

7. API спецификация

  1. REST API endpoints
    • http
    • POST /api/rooms/create - Создание новой комнаты
    • GET /api/rooms/:id/status - Проверка статуса комнаты
    • POST /api/rooms/:id/join - Присоединение к комнате
  2. WebSocket события
// Клиент → Сервер
client.emit('join-room', { roomId, userData })
client.emit('signal', { to, signal })

// Сервер → Клиент
server.emit('user-joined', { userData })
server.emit('signal', { from, signal })

8. Безопасность

  1. Меры защиты
    Валидация входных данных
    Ограничение размера комнат
    HTTPS/WSS соединения
    Sanitization сообщений чата
  2. Защита приватности
    Шифрование медиапотокков (DTLS-SRTP)
    Отсутствие записи разговоров
    Автоматическое удаление данных сессии

9. Этапы разработки

  1. Этап: Базовый функционал (4 недели)
    • WebRTC соединение 1 на 1
    • Базовая верстка интерфейса
    • Сигналинг сервер
  2. Этап: Групповые звонки (3 недели)
    • Mesh сеть до 12 участников
    • Интерфейс сетки видео
    • Управление аудио/видео
  3. Этап: Дополнительные функции (3 недели)
    • Демонстрация экрана
    • Текстовый чат
    • Настройки устройств
  4. Этап: Оптимизация и тестирование (2 недели)
    • Тестирование производительности
    • Исправление ошибок
    • Документация

10. Тестирование

  1. Типы тестов
    • Unit тесты: Jest для логики компонентов
    • Integration тесты: WebRTC соединения
    • E2E тесты: Cypress для сценариев использования
  2. Критерии качества
    • 95% покрытие кода тестами
    • Задержка аудио < 200 мс
    • Время подключения < 5 секунд
    • Поддержка 12 участников без лагов

11. Развертывание

  1. Инфраструктура
    • Frontend: Vercel/Netlify
    • Backend: Node.js сервер (Heroku/DigitalOcean)
    • WebRTC: STUN/TURN сервер (Coturn)
  2. Мониторинг
    • Логи ошибок (Sentry)
    • Метрики производительности
    • Мониторинг uptime

12. Ограничения и будущие улучшения

  1. Текущие ограничения
    • Максимум 12 участников на комнату
    • Нет записи встреч
    • Нет интеграции с календарями
    • Без системы аутентификации
  2. Возможные улучшения
    • SFU архитектура для больших групп
    • Запись встреч
    • Виртуальный фон
    • Поднятие руки
    • Комнаты для переговоров

Приложения:

  • Макеты интерфейса (Figma)
  • Диаграмма архитектуры
  • Чек-лист тестирования
  • Сроки: 12 недель

Команда: 2 фронтенд, 1 бэкенд разработчик, 1 тестировщик
